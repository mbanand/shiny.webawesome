## Discussions and decisions

* We chose `wa_js()` as a narrow helper that solves the app-structure problem
  of getting small JavaScript snippets onto the page without building a custom
  event DSL or a broader JS asset system.
* We explicitly kept `wa_js()` string-based rather than teaching it about file
  serving or bundling. Reading JS from a file remains a user-level R workflow
  that produces the string passed into `wa_js()`.
* We treated browser-local event plumbing and property reads as documentation
  and usage-pattern problems rather than new package-API surface.
* We decided not to add all standard/global HTML attrs to generated wrappers.
  Instead we adopted a narrow package-level exception for `class` and `style`
  only.
* We required duplicate suppression when injecting `class` and `style` so a
  future upstream metadata change does not create duplicated wrapper args.
* We kept the package-level rule explicit in architecture docs because this is
  a generator policy choice, not something inferred from current metadata.
* We also refreshed two stale integrity-summary test strings during validation
  so the tool tests match the current integrity-check wording.

## Proposed next steps

* Fold `wa_js()`, `wa_set_property()`, and `wa_call_method()` into a later
  user-facing advanced browser-glue guide rather than adding a one-off page in
  isolation.
* Add user-facing documentation explaining that attributes beyond the
  generated wrapper surface can still be attached with `htmltools` helpers
  rather than by expanding the generator to cover arbitrary global attrs.

## Pitfalls, reminders, and pointers

* `wa_js()` is intentionally small. It should stay focused on emitting inline
  JavaScript strings and should not quietly expand into an asset-management or
  event-helper subsystem.
* The current property-read guidance is documentation-based and relies on
  browser-side reads plus `Shiny.setInputValue()`, not a new package API.
* Generated `class` and `style` are package-level convenience attrs, not
  upstream metadata claims. If upstream eventually adds one of them to a
  component, the generator should continue surfacing only one wrapper arg.
* Attributes beyond the generated wrapper surface remain a user-space
  `htmltools` story for now and should not be treated as an invitation to add
  more blanket generator exceptions without a separate policy decision.
