## Discussions and decisions

* We agreed that the minimal Phase D3 `shinytest2` slice was worth adding before broader generation because the representative surface was now stable enough to justify one real browser/runtime validation path.
* We agreed that the broad behavioral test files under `tests/testthat/` are acceptable for now because they are organized by current surface/behavior rather than by one file per generated component. We explicitly treated that as appropriate for the current representative phase.
* We confirmed that the CRAN testing policy is being followed correctly:
  browser tests use `skip_on_cran()`
  browser tests also skip cleanly when no Chromium-based browser is available.
* We agreed that once the browser test proved the runtime issue concretely, the correct upstream browser-runtime source is `dist-cdn/`, not `dist/`.
* We then explicitly tightened the pipeline to reflect that discovery:
  this is no longer a compatibility preference
  it is now an explicit repository rule in practice:
  runtime fetch/prune work off upstream `dist-cdn/`.
* We also agreed that vendoring upstream `dist-cdn/` under a local path named `dist/` would be confusing and should be cleaned up immediately once the correct source was proven. The repository now uses `vendor/webawesome/<version>/dist-cdn/` honestly.
* The browser test directly uncovered two generator bugs in the binding template, which validates the decision to add Phase D3 before broader generation:
  native subscription callbacks must not forward raw DOM events into Shiny
  server-side update paths must dispatch real DOM events, not jQuery-only synthetic events, because the generated bindings subscribe with native listeners.

## Pitfalls, reminders, and pointers

* The package-source layout remains the standard top-level `R/` layout. Do not reintroduce `R/core/`, `R/generated/`, or `R/generated_updates/`.
* Generated package files under top-level `R/` are still generated outputs. Do not hand-edit generated component files.
* The browser/runtime source of truth for vendored upstream runtime is now `vendor/webawesome/<version>/dist-cdn/`. Do not silently reintroduce `vendor/.../dist/` assumptions in fetch, prune, tests, or docs.
* The representative browser/runtime test now matters operationally. It already caught:
  upstream runtime-source mistakes,
  a generated binding subscription bug,
  and a generated receiveMessage event-dispatch bug.
  Keep using it as a real validation gate while broadening generation.
* The generated binding template now relies on native DOM events consistently:
  subscribe uses a wrapped native listener
  receiveMessage dispatches a real bubbling `change` event.
  Preserve that consistency if binding logic evolves.
* The generated `update_wa_select()` helper remains colocated with `wa_select()` and documented with `@rdname wa_select`. Continue that pattern where it keeps one component API together.
* The support-model tightening work from the earlier session is still only a first pass. Obvious SSR/internal wrapper attributes were excluded, but the broader user-facing policy still needs deliberate review across more components before large-scale generation expansion.
