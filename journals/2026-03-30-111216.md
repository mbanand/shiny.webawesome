## Discussions and decisions

* We settled the checksum-hardening design as stage-owned integrity records
  rather than ad hoc per-stage checksum snippets.
* We kept stage-level integrity behavior advisory so developers can continue
  through intermediate debug states, but added a dedicated final
  `check_integrity.R` gate that fails during `build_package`.
* We excluded `inst/www/webawesome-init.js` from the prune-owned integrity
  surface because it is handwritten package glue rather than prune output.
* We kept integrity records in `manifests/integrity/` as generated build
  diagnostics, separate from package assets and from human-readable reports.
* We retained both per-file digests and an overall tree digest so mismatches
  are diagnosable rather than opaque.
* The distclean/build failure investigation showed that the real repository
  file `R/wa_warning_registry.R` was not missing; a fake repo fixture in
  `test-review_binding_candidates.R` was missing it.

## Pitfalls, reminders, and pointers

* The integrity helper logic should remain centralized in `tools/integrity.R`.
  Reimplementing tree hashing separately inside individual stages would make
  drift likely.
* The final integrity gate only works as intended if stage-owned surfaces stay
  explicit and narrow. Do not casually widen them to include handwritten files
  or unrelated diagnostics.
* Distclean-safe orchestration depends on fake repo fixtures creating any
  handwritten helper files that shared generator/report helpers now require.
  If future shared helper requirements expand, update the tool-test fixtures
  deliberately rather than weakening the runtime checks.
