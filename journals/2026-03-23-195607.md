## Discussions and decisions

* We confirmed that `wa-button` exposed the first concrete need for a formal generator policy seam:
  upstream metadata is still the primary source of truth, but it is not sufficient for every Shiny support-model decision.
* We agreed that `wa-button` should not be forced into the value-input binding model.
  The correct behavior is an explicit action-binding mode using Shiny action semantics.
* We agreed the override seam should stay narrow and policy-driven, not turn into a broad handwritten component rewrite mechanism.
* We agreed the candidate-discovery pass should be advisory and conservative:
  it should help find likely metadata-gap review targets, not silently auto-classify them.
* We agreed to leave heuristic tuning for near-miss candidate discovery to the next session rather than broadening the review signals immediately at session close.
* We also agreed to keep the repository in its current generated state:
  all 58 components are presently generated, but full test coverage has not yet caught up with that broader generated surface.

## Proposed next steps

* Tune the binding-candidate discovery heuristics and inspect near-miss components manually to determine whether any additional explicit support-model overrides are warranted.
* Decide whether the candidate-discovery report should stay conservative or gain a second, broader “near misses / watch list” section for human review.
* Reconcile the repository’s current 58-component generated surface with test coverage completeness, keeping clear track of the fact that the generated surface is ahead of the most complete focused representative test coverage.

## Deferred items

These remain deferred and should stay visible in the next session.

* Manual override layer deferment, now narrowed:
  keep the override seam small and policy-driven rather than turning it into a broad handwritten component customization layer.
  The first real use case now exists (`wa-button`), but that does not justify a general-purpose override architecture.
* Clean-stage structural revisit deferment:
  the previously noted clean-stage structural question remains open and should not silently disappear from the mental model just because other generator and runtime work progressed.
  The current clean implementation still works, but the broader long-term shape of clean-stage structure versus earlier runner/stage patterns is not formally closed.
* Checksum hardening deferment:
  checksum-based integrity validation across prune and generate remains deferred.
  It is still a useful hardening direction, but it should wait until the broader generated surface and stage interfaces settle further.
* Manifest/reporting deferment:
  generated-file manifests and broader coverage/conformance reporting remain deferred until the broader generated package surface is more stable.
* Custom-event helper deferment:
  a future opt-in client-side helper layer for attaching focused component-specific JavaScript event handlers remains deferred.
  It should stay clearly separate from the Shiny server-input binding model.
* Component-method exposure deferment:
  future support for upstream component methods remains deferred.
  If it is added later, it should remain separate from reactive input bindings and likely split into a server-side command layer plus a client-side helper layer, with selective method exposure rather than blanket automatic generation.
* Coverage completeness deferment:
  the repository now contains the broader 58-component generated surface, but the most complete focused representative test coverage still reflects the earlier 25-component broadening pass plus the targeted `wa-button` action-binding work.
  Keep this discrepancy visible in future review and validation decisions; do not casually treat the broadened generated surface as if test completeness has already caught up.

## Pitfalls, reminders, and pointers

* `wa-button` is now a policy-driven action-bound component, not a wrapper-only component and not a value-input binding.
  Preserve that distinction in future generator, wrapper, and runtime changes.
* The new binding-candidate report currently uses a conservative threshold.
  “No additional candidates” means “none above the current heuristic bar,” not “proof that no other metadata gaps exist.”
* The review tool is meant to aid human judgment, not replace it.
  Keep it advisory and deterministic.
* The repository is currently ahead of full test coverage completeness because the broader 58-component generated surface is present while focused test coverage is still narrower.
  Keep that asymmetry visible in planning, review, and future journal entries.
