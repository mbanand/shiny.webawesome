* Improved the human-readable component conformance report format so future
  nonconformant components are rendered as compact per-component blocks that

## Discussions and decisions

* We explicitly documented that behavioral tests and structural conformance
  reporting solve different problems and should both remain part of the
  project.
* We kept the final conformance deepening pass structural rather than
  behavioral: it checks deterministic emitted wrapper and binding internals,
  but does not try to simulate runtime behavior or duplicate browser tests in
  the report stage.
* We kept the report-side oracle generator-derived by reusing shared render
  helpers wherever possible.
* We treated the reporting stage as effectively covered for the current scope.
  Any next work is more about project direction and CI policy than obvious
  missing report-stage plumbing.

## Pitfalls, reminders, and pointers

* The structural comparisons should stay generator-derived. If future report
  logic starts restating wrapper or binding behavior independently, the
  conformance layer will drift from generation again.
* Slot-call extraction should continue to compare unnamed ordered vectors.
  Named vectors from helper extraction can produce false mismatches even when
  the emitted slot calls are identical.
* The markdown conformance report should continue to show only failed areas
  for nonconformant components. Rendering every field on one line makes
  failure reports difficult to scan.
