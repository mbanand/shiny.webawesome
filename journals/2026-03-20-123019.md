## Discussions and decisions

* We agreed that generated-file manifests and reporting remain deferred until later in the pipeline. They should not be introduced during the early generate implementation because the emitted package surface is still intentionally in flux.
* We agreed on the milestone plan for generate: metadata parsing and extraction, intermediate schema, minimal handwritten runtime, templates, partial generation with representative components, manual override seam, full generation, broader test/document/orchestrator wiring, lint/style cleanup, and final full-pipeline execution.
* We agreed to keep the intermediate schema broad for now, even when the upstream metadata includes public members that look internal from an R/Shiny perspective. Tightening should happen later in the generated support model, not by discarding upstream facts prematurely from the intermediate representation.
* We agreed that generate should fail early and gracefully when required prune outputs are missing, and that it should tailor the guidance based on whether upstream fetched assets exist under `vendor/`. That was implemented in this session.
* We discussed checksum validation of prune outputs and agreed it is a good hardening idea, but should be deferred until later because it spans both prune and generate and is not required to keep current generate work moving.
* We agreed on representative components for the first generator pass after clarifying the distinction between event support and value bindings: `wa-card` for wrapper only, `wa-checkbox` for wrapper plus binding, and `wa-select` for wrapper plus binding plus update helper. `wa-button` remains relevant for event handling later but is not the right representative for the value-binding category.
* We agreed that `wa_page()` should remain narrowly scoped in the first pass: it should provide package-level dependency attachment and page-level affordances needed for Shiny usage and examples, but it should not become a large layout abstraction.
* We agreed that the package must support both usage patterns: package-native assembly through `wa_page(...)` and component-by-component use inside standard Shiny containers such as `fluidPage(...)`. The handwritten runtime and tests now reflect that expectation.
* We agreed that `wa_page()` is a justified and explicit exception to the normal upstream-mirroring rule because it follows Shiny page semantics rather than wrapping the upstream Pro `wa-page` component. That distinction must remain prominently documented to avoid confusion if upstream `wa-page` support is ever added later.
* We agreed on the testing policy for generated wrappers: assert exact rendered component fragments for wrapper output, test page/dependency/runtime behavior separately, and reserve `shinytest2` for browser/runtime integration rather than raw markup snapshots. Upstream markdown references under `vendor/.../dist/skills/webawesome/references/` may be consulted as a secondary aid, but the primary oracles remain upstream metadata plus our own generator mapping rules.
* We agreed that `--debug`, `--filter`, and `--exclude` belong in the initial generate CLI contract rather than as late additions, because they are most useful during schema and partial-generation development. We also agreed to use `--exclude` rather than a less readable alternative like `--filterex`.
* We agreed that style and lint validation should happen incrementally during generate implementation rather than being postponed to a final cleanup stage, because that keeps churn smaller and makes generator/template problems easier to isolate.
