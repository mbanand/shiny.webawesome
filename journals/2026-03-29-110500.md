## Discussions and decisions

* We agreed to stop treating reporting as deferred and to implement it now as
  the last stage of the current pipeline.
* We kept report separate from generate because generation and verification are
  distinct concerns in the architecture and the docs already describe them
  that way.
* We implemented reporting in an initial artifact-first form for conformance:
  wrapper presence, export presence, expected binding presence, and expected
  update-function presence/export state relative to the selected support
  model.
* We accepted that deeper per-attribute, per-property, and per-event
  conformance should come later, but it is now an iterative enhancement on top
  of a working report stage rather than a blocker to having report at all.

## Proposed next steps

* Broaden component API conformance beyond artifact presence to include more
  detailed argument/property/event/slot alignment checks against upstream
  metadata.
* Decide how strict CI should be for report outputs now that the stage exists,
  especially around whether any future partial or unsupported component should
  fail or only warn.
* Add iterative passes across generate -> test -> report to tighten the
  current surface now that a deterministic report baseline exists.

## Deferred items

These remain deferred and should stay visible in the next session.

* Checksum hardening deferment:
  checksum-based integrity validation across prune and generate remains
  deferred.
  It is still a useful hardening direction, but it should wait until the
  broader generated surface and stage interfaces settle further.

* Custom-event helper deferment:
  a future opt-in client-side helper layer for attaching focused
  component-specific JavaScript event handlers remains deferred.
  It should stay clearly separate from the Shiny server-input binding model.

* Component-method exposure deferment:
  future support for upstream component methods remains deferred.
  If it is added later, it should remain separate from reactive input
  bindings and likely split into a server-side command layer plus a
  client-side helper layer, with selective method exposure rather than
  blanket automatic generation.

* Live-property command-layer deferment:
  future support for writable live component properties that are distinct
  from constructor-time HTML attributes remains deferred.
  If added later, it should stay separate from generated wrapper constructor
  arguments and live in a distinct server/client command or update layer.

* User-facing binding-convention documentation deferment:
  user-facing documentation for the Shiny binding convention remains
  deferred.
  Future documentation should explain explicitly that action-like controls
  may bind as actions, action-with-payload controls such as `wa-dropdown`
  split the action trigger from the latest payload side channel, and other
  bindings may expose durable semantic state rather than raw browser event
  names.

* Semantic binding surface documentation deferment:
  user-facing documentation for the current semantic binding surface remains
  deferred.
  The current points that need explicit documentation are the native 0-based
  `wa-carousel` value contract, the selected-item-id contract plus
  id-omission warning behavior for `wa-tree`, the boolean `open` contracts
  for `wa-details`, `wa-dialog`, and `wa-drawer`, the active-tab-name
  contract for `wa-tab-group`, and the ordinary durable-value behavior now
  covered by the dedicated semantic harness apps.

* Warning and user-option documentation deferment:
  the package warning registry and related user-facing runtime options remain
  underdocumented for users and maintainers beyond code comments and runtime
  warnings.
  Future documentation should explain
  `options(shiny.webawesome.warnings = list(...))`, the currently known
  warning keys and suppression semantics, and the optional
  `options(shiny.webawesome.runtime_harness.log = TRUE)` development-only
  harness logging flag.

## Pitfalls, reminders, and pointers

* `tools/report_components.R` deliberately loads the shared generate helpers
  lazily. If future changes move helper sourcing back to file-load time, the
  tool-documentation workflow will regress.
* The current conformance manifest is intentionally artifact-first, not yet a
  full argument/property/event/slot-level checker. Future work should deepen
  it rather than replace it with ad hoc manual notes.
* The report stage now owns the manifest-writing step. Do not drift back to
  describing manifests as generate-stage artifacts in the docs.
