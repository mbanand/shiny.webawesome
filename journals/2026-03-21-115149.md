## Discussions and decisions

* We agreed to keep the API distinction between wrapper-only DOM ids and Shiny input ids explicit:
  wrapper-only components use `id`,
  Shiny-bound components use `input_id`,
  and `input_id` is the first positional argument for bound/generated components.
* We agreed not to invent a separate second DOM-id concept for bound components at this stage.
  `input_id` is the user-facing Shiny identifier, and that value is also written to the rendered DOM `id` attribute.
  This remains compatible with DOM-targeting utilities such as `shinyjs` because they operate on the actual rendered DOM `id`.
* We agreed to document the API distinction at the generator/source level rather than by hand in individual generated files.
  Generated roxygen now carries that distinction automatically.
* We agreed that the previously documented post-generate `styler`/`lintr` expectation needed to be made explicit and operational, because the prior session had allowed raw template output to persist without a final style pass.
* We agreed that the debug schema artifact should optimize for humans because that is its operational purpose.
  That led to changing the debug `components` collection from an unnamed array to a sorted tag-keyed map.
* We agreed that broader generation at this stage should primarily be used to pressure-test the support model and generator, not to rush toward blanket upstream coverage.
* We agreed to broaden through a deliberately chosen representative batch rather than an indiscriminate larger set.
  The chosen batch maximized variation across:
  wrapper-only slot/attribute richness,
  binding-only value/event patterns,
  and update-capable value/control surfaces.
* We agreed to replace the earlier hardcoded support model with a rule-based heuristic model.
  The first implementation is conservative by design.
  That was an important choice:
  over-classification would have produced a misleadingly large generated reactive surface without enough confidence that the generic bindings and update helpers were semantically correct.
* We agreed that update classification should not simply mean “component has a state-looking field”.
  The implemented heuristic therefore requires both a bindable event surface and a richer writable state/value-control surface before a component is promoted to update-capable.
* We explicitly checked whether the broadened pass revealed a need for a manual override seam.
  At the end of this session, the answer remains no:
  the broadened work exposed support-model and generator/template issues, but not a concrete component-specific situation that presently requires a formal override architecture.

* Generated package files under top-level `R/` are still generated outputs. Do not hand-edit generated component files.

* The debug schema artifact now serves a genuine operational role in support-model review.
  Keep it human-readable:
  `components` is now a sorted tag-keyed map and the `classification.reasons` block is meaningful during review.
* The current broadened generate support is classification-driven, not hardcoded by tag.
  Future support changes should prefer refining the heuristic lists and generic renderer behavior before falling back to tag-specific branching.
* The broadened browser/runtime harness now matters operationally. It validates:
  wrapper-only rendering,
  binding-only propagation for additional controls,
  and update-helper propagation for additional controls.
  Keep it representative and disciplined.
* `input_id` versus `id` is now an explicit architectural rule:
  wrapper-only components use `id`,
  bound components use `input_id`,
  and `input_id` is the first positional argument for bound components.
  Preserve that distinction in future generator work and generated docs.
