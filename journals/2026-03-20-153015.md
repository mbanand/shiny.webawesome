## Discussions and decisions

* We agreed that a speculative manual override seam should still remain deferred. The generator work so far has exposed support-model policy decisions, not a concrete architectural gap that requires an override layer right now.
* We agreed that generated package API documentation is the right next concern before broadening generation, but that it should stay minimal for now: short generated roxygen, exports, and man pages, with more polished examples and long-form docs deferred until the public surface is more stable.
* We explicitly corrected an implementation misstep: the temporary roxygen shim approach was removed. Documentation now lives with generated code again, in line with roxygen philosophy.
* We agreed that the right package layout is the standard R package convention: no subdirectories under `R/`; all package source files, handwritten and generated, live directly under top-level `R/`.
* We agreed to use `.impl`-style split files sparingly, and if they are needed, they should still live at top-level `R/`. After re-evaluating `wa_page()`, we agreed it was small enough to collapse into a single top-level file.
* We confirmed the practical reason the reorganization was necessary: in this package setup, standard `roxygen2::roxygenise()` only picked up the top-level `R/*.R` package files. Once generated component files moved to top-level `R/`, standard roxygen behavior worked as expected.
* We agreed that wrapper/update pairs should be colocated where appropriate and documented together. The representative `wa_select()` / `update_wa_select()` file now follows that model.
* We agreed that after the layout reorganization, the project docs and `AGENTS.md` had to be updated immediately rather than leaving them stale, because many of them were still describing `R/core`, `R/generated`, and `R/generated_updates`.

## Pitfalls, reminders, and pointers

* The package-source layout has changed materially. All package source now lives directly under top-level `R/`. Do not reintroduce `R/core/`, `R/generated/`, or `R/generated_updates/` without a very strong reason; that would recreate the package-tooling friction we just removed.
* Generated package files are still generator outputs even though they now live directly under `R/`. Do not hand-edit generated component files. Fix generator logic/templates and regenerate instead.
* The debug output path fix is important for human review: `--debug` now writes persistent artifacts under `scratch/debug/`, and those snapshots should be used in the next support-model review pass.
* The support-model tightening done this session is only a first pass. It currently excludes obvious SSR/internal wrapper attributes, but the broader user-facing policy still needs deliberate review across more upstream components.
* The representative package surface is now on standard package tooling, which is a major simplification. Preserve that simplicity when broadening generation.
* The generated `update_wa_select()` helper is now colocated with `wa_select()` and documented with `@rdname wa_select`. Continue that pattern where it keeps a component API together.
* The unresolved carry-forward note remains active:
  keep the deferred clean-stage structural question visible until it is actually resolved, not merely overshadowed by later progress. The current clean implementation now supports top-level generated `R/*.R` cleanup by generated-file header, but the broader question of the long-term clean-stage structure versus earlier runner/stage patterns is still not formally closed.
* The prune/generate checksum validation idea remains deferred:
  checksum-based integrity validation across prune and generate is still a good hardening direction, but it should wait until the broader generated surface and pipeline interfaces settle further.
