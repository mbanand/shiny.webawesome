## Discussions and decisions

* We agreed to reject `wa-copy-button` as a semantic binding candidate and not
  pursue a generic `wa-error` binding family because the current `wa-error`
  cases are client-side notification/error signals without an obvious default
  server-reactive contract.
* We agreed that `wa-tab-group` belongs to the semantic binding family rather
  than the open-state family, and that its Shiny contract should be active tab
  name rather than raw `wa-tab-show` or `wa-tab-hide` events.
* We agreed that `wa-details`, `wa-dialog`, and `wa-drawer` should expose
  committed boolean `open` state to Shiny.
* We agreed that the binding convention should be stated explicitly:
  action-like controls may bind as actions, while other bindings should expose
  the durable semantic value implied by the relevant browser event.
* We agreed that lifecycle-driven semantic state should use the earliest
  non-cancelable events that commit the state, which led to
  `wa-details: wa-show/wa-hide`,
  `wa-dialog: wa-show/wa-after-hide`,
  and `wa-drawer: wa-show/wa-after-hide`.
* We agreed that the package should not rewrite user input ids to add a `wa_`
  prefix automatically; app authors who want explicit namespacing should
  choose prefixed `input_id` values themselves.
* We agreed that internal semantic extractor keys may continue mirroring
  upstream property names such as `activeSlide` because they are not currently
  user-facing Shiny naming contracts.

## Deferred items

These remain deferred and should stay visible in the next session.

* Clean-stage structural revisit deferment:
  the previously noted clean-stage structural question remains open and
  should not silently disappear from the mental model just because other
  generator and runtime work progressed.
  The current clean implementation still works, but the broader long-term
  shape of clean-stage structure versus earlier runner/stage patterns is not
  formally closed.

* Checksum hardening deferment:
  checksum-based integrity validation across prune and generate remains
  deferred.
  It is still a useful hardening direction, but it should wait until the
  broader generated surface and stage interfaces settle further.

* Manifest/reporting deferment:
  generated-file manifests and broader coverage/conformance reporting remain
  deferred until the broader generated package surface is more stable.

* Custom-event helper deferment:
  a future opt-in client-side helper layer for attaching focused
  component-specific JavaScript event handlers remains deferred.
  It should stay clearly separate from the Shiny server-input binding model.

* Component-method exposure deferment:
  future support for upstream component methods remains deferred.
  If it is added later, it should remain separate from reactive input
  bindings and likely split into a server-side command layer plus a
  client-side helper layer, with selective method exposure rather than
  blanket automatic generation.

* Live-property command-layer deferment:
  future support for writable live component properties that are distinct
  from constructor-time HTML attributes remains deferred.
  If added later, it should stay separate from generated wrapper constructor
  arguments and live in a distinct server/client command or update layer.

* Coverage completeness deferment:
  the repository now contains the broader generated component surface, but the
  most complete focused representative test coverage still trails the full
  breadth of that surface.
  Keep this discrepancy visible in future review and validation decisions;
  do not casually treat the broadened generated surface as if test
  completeness has already caught up.

* User-facing binding-convention documentation deferment:
  user-facing documentation for the Shiny binding convention remains deferred.
  Future documentation should explain explicitly that action-like controls may
  bind as actions, while other bindings expose the durable semantic value
  implied by the relevant browser event rather than the raw event name.

* Semantic binding surface documentation deferment:
  user-facing documentation for the current semantic binding surface remains
  deferred.
  The current points that need explicit documentation are the native 0-based
  `wa-carousel` value contract, the selected-item-id contract plus
  id-omission warning behavior for `wa-tree`, the boolean `open` contracts
  for `wa-details`, `wa-dialog`, and `wa-drawer`, and the active-tab-name
  contract for `wa-tab-group`.

* Warning registry documentation deferment:
  the package warning registry mechanism remains underdocumented for users
  and maintainers beyond code comments and generated warnings.
  Future documentation should explain
  `options(shiny.webawesome.warnings = list(...))`, the currently known
  warning keys, and the intended suppression semantics.

## Pitfalls, reminders, and pointers

* The semantic binding convention is now explicit in the architecture docs,
  but the user-facing docs still need to explain that raw browser event names
  are not always the Shiny contract.
* `wa-dialog` and `wa-drawer` intentionally use `wa-after-hide` rather than
  `wa-hide` for the close-side commit signal because the earlier hide event is
  request-like and cancelable.
* `wa-tab-group` now exposes active panel name via `active`.
  That should continue to be treated as selection-state, not as an open-state
  special case, in future discussions.
