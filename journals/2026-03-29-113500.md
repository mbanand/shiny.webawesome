## Discussions and decisions

* We kept the conformance deepening focused on deterministic generated-surface
  checks rather than jumping immediately to brittle full-code parsing or a
  large handwritten conformance exception layer.
* We explicitly kept the report helper bootstrap lazy. The tool-doc workflow
  still depends on being able to load `report_components.R` in isolation.
* We agreed that report-runner summary wording should use the actual coverage
  status categories (`covered`, `partial`, `planned`, `excluded`,
  `unsupported`) rather than `coverage_*` names.

## Proposed next steps

* Deepen wrapper conformance further to compare more emitted wrapper internals,
  such as normalized attribute maps, boolean-name maps, and slot emission.
* Deepen binding conformance further to compare more emitted behavior inside
  `getValue()` and `receiveMessage()` for selected support modes.
* Prefer a generator-shared-helper approach for that deeper pass: factor
  expected emitted internals out of the render layer so report can compare
  against generator-derived structures directly rather than growing a second,
  parser-heavy interpretation of the generated code.
* Treat that next pass as an involved `generate -> test -> report` cycle,
  because it will touch generator/render logic as well as reporting.
* Decide the stopping point for conformance deepening before implementing it.
  The current expectation is that the next pass should likely be the last major
  conformance deepening step rather than the start of an open-ended expansion.
* Start deciding which of the report outputs should become CI gates versus
  informational diagnostics as the conformance layer continues to broaden.

* The deeper conformance expectations should stay generator-derived. If future
  changes reimplement wrapper/update/binding expectations independently inside
  report, the conformance layer will drift from generate again.

  definitions both need to remain parseable by the report stage.
