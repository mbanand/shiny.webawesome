## Discussions and decisions

* We agreed to keep omitted-native-event review narrowly focused on likely
  missing action semantics rather than broadening to a generic browser-native
  event inventory.
* We agreed that the `review_binding_candidates` tool should separate
  high-confidence candidates from a near-miss watch list, and that the tool
  should remain advisory rather than becoming an auto-classifier.
* We agreed that the watch-list components found in this session did not
  justify new omitted-native-event overrides, including `wa-copy-button`,
  `wa-option`, `wa-radio`, `wa-tab`, and `wa-tab-group`.
* We agreed to treat semantically meaningful component events as a separate
  curated semantic binding category rather than extending `.binding_event_names()`
  with regex rules such as `wa-*-change`.
* We agreed to keep show/hide lifecycle work out of the first semantic binding
  pass and, if revisited later, to treat it as an open-state binding question
  rather than as omitted-native-event recovery.
* We agreed that `wa-carousel` should expose the native 0-based
  `activeSlide` value to Shiny rather than translating it to 1-based R
  indexing in the first implementation.
* We agreed that `wa-tree` should use selected item DOM ids as its first-pass
  Shiny contract, and that silently dropping id-less selected items would be
  too surprising without a runtime warning.
* We agreed to implement the warning suppression mechanism as a list-based
  package warning registry instead of a one-off scalar option, because the
  browser runtime needs structured access to it and the mechanism will likely
  remain useful long term.
* We agreed that warning behavior should remain declarative in policy via
  known warning keys rather than arbitrary inline warning code.

## Proposed next steps

* Decide whether to pursue additional curated semantic bindings beyond

## Deferred items

These remain deferred and should stay visible in the next session.

* Manual override layer deferment, now narrowed:
  keep the override seam small and policy-driven rather than turning it into
  a broad handwritten component customization layer.
  The first real use case now exists (`wa-button`), but that does not justify
  a general-purpose override architecture.
* Clean-stage structural revisit deferment:
  the previously noted clean-stage structural question remains open and
  should not silently disappear from the mental model just because other
  generator and runtime work progressed.
  The current clean implementation still works, but the broader long-term
  shape of clean-stage structure versus earlier runner/stage patterns is not
  formally closed.
* Checksum hardening deferment:
  checksum-based integrity validation across prune and generate remains
  deferred.
  It is still a useful hardening direction, but it should wait until the
  broader generated surface and stage interfaces settle further.
* Manifest/reporting deferment:
  generated-file manifests and broader coverage/conformance reporting remain
  deferred until the broader generated package surface is more stable.
* Custom-event helper deferment:
  a future opt-in client-side helper layer for attaching focused
  component-specific JavaScript event handlers remains deferred.
  It should stay clearly separate from the Shiny server-input binding model.
* Component-method exposure deferment:
  future support for upstream component methods remains deferred.
  If it is added later, it should remain separate from reactive input
  bindings and likely split into a server-side command layer plus a
  client-side helper layer, with selective method exposure rather than
  blanket automatic generation.
* Live-property command-layer deferment:
  future support for writable live component properties that are distinct
  from constructor-time HTML attributes remains deferred.
  If added later, it should stay separate from generated wrapper constructor
  arguments and live in a distinct server/client command or update layer.
* Coverage completeness deferment:
  the repository now contains the broader 58-component generated surface,
  but the most complete focused representative test coverage still reflects
  the earlier broadening pass plus the targeted `wa-button` action-binding
  and follow-on cleanup work.
  Keep this discrepancy visible in future review and validation decisions;
  do not casually treat the broadened generated surface as if test
  completeness has already caught up.
* Open-state semantic binding deferment:
  show/hide lifecycle binding review remains deferred as a separate
  open-state question rather than omitted-native-event recovery.
  The one-off candidate set identified this session was:
  `wa-color-picker`, `wa-tooltip`, `wa-details`, `wa-dialog`, `wa-drawer`,
  `wa-dropdown`, `wa-select`, and `wa-popover`.
  Current thinking to preserve:
  `wa-details` maybe yes, `wa-drawer` maybe yes, `wa-dialog` possible,
  `wa-tooltip`/`wa-popover` probably no as defaults, and
  `wa-select`/`wa-dropdown` probably no as defaults.
* Broader semantic-event expansion deferment:
  curated semantic binding expansion beyond `wa-carousel` and `wa-tree`
  remains deferred.
  The components that still look most relevant for later review are
  `wa-copy-button` and `wa-tab-group`, and they should remain framed as
  semantic-event candidates rather than omitted-native-click cases.
* Semantic binding documentation deferment:
  user-facing documentation for the semantic binding surface remains
  deferred.
  The current points that need explicit documentation are the native 0-based
  `wa-carousel` value contract and the selected-item-id contract plus
  id-omission warning behavior for `wa-tree`.
* Warning registry documentation deferment:
  the new package warning registry mechanism remains undocumented for users
  and maintainers beyond code comments and generated warnings.
  Future documentation should explain
  `options(shiny.webawesome.warnings = list(...))`, the currently known
  warning keys, and the intended suppression semantics.

## Pitfalls, reminders, and pointers

* `wa-carousel` now binds the native 0-based `activeSlide` property.
  That is intentional, but it is exactly the sort of detail that will confuse
  R users if it is not called out explicitly in future documentation.
* `wa-tree` currently exposes selected DOM ids only.
  If selected items do not have ids, the binding omits them and relies on the
  warn-once helper to make that loss visible; this behavior should not be
  forgotten when evaluating whether the first-pass contract remains adequate.
* The warning registry/config bootstrap currently lives in the package
  dependency path, not in generated component-specific code.
  That is the intended reusable seam for future warning-backed bindings.
