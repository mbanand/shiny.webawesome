Important validation observations from this session:

* Several durable-value bindings are commit-driven on the Shiny side in the
  harness apps, so the browser tests should begin from the first deliberate
  committed interaction rather than assume that constructor-time default
  values appear immediately in `input$...`.
* `wa-number-input`, `wa-slider`, and `wa-rating` reach Shiny as numeric
  values rather than strings in these browser tests, so the helper logic and
  visible harness state formatting should preserve numeric comparison rather
  than coerce through string-only expectations.
* The real `wa-tree` warning contract in the browser is:
  selected missing-id items are omitted from the Shiny value, the warning is
  emitted once per input id, and a selection containing only omitted items
  yields `NULL` rather than an empty character vector.
* The presentational/runtime-presence harness works as intended as a manual
  inspection surface without implying any default Shiny binding contract for
  those components.

## Discussions and decisions

* We agreed that the browser-runtime layer should mirror the wrapper-test
  ownership model at the file level, but not at the app level. The durable
  test structure is now component-owned test files plus a small number of
  shared harness apps grouped by contract type.
* We agreed that the harness apps should be genuinely useful manual
  inspection surfaces, not just machine fixtures. That drove the centered
  titles, alphabetical linked indexes, alphabetical sections, short
  human-facing descriptions, visible observed state blocks, and optional
  controlled logging.
* We agreed to keep console/server logging useful but optional and not make
  it part of the core browser contract.
* We agreed that `wa-dropdown` remains the only action-style /
  action-with-payload contract in the current generated browser surface and
  therefore warranted its own dedicated harness app.
* We agreed that `wa-tree` is its own category because of the selected-id
  vector contract plus the missing-id warning path, rather than just another
  ordinary semantic-value input.
* We agreed that `wa-rating` does not currently have a project-specific
  quirk category beyond being a normal durable numeric value input.
* We agreed that the remaining non-bound components should share one final
  presentational/runtime-presence harness app, but still follow the same
  formatting and manual-inspection conventions as the bound harnesses.

## Deferred items

These remain deferred and should stay visible in the next session.

* Checksum hardening deferment:
  checksum-based integrity validation across prune and generate remains
  deferred.
  It is still a useful hardening direction, but it should wait until the
  broader generated surface and stage interfaces settle further.

* Manifest/reporting deferment:
  generated-file manifests and broader coverage/conformance reporting remain
  deferred until the broader generated package surface is more stable.

* Custom-event helper deferment:
  a future opt-in client-side helper layer for attaching focused
  component-specific JavaScript event handlers remains deferred.
  It should stay clearly separate from the Shiny server-input binding model.

* Component-method exposure deferment:
  future support for upstream component methods remains deferred.
  If it is added later, it should remain separate from reactive input
  bindings and likely split into a server-side command layer plus a
  client-side helper layer, with selective method exposure rather than
  blanket automatic generation.

* Live-property command-layer deferment:
  future support for writable live component properties that are distinct
  from constructor-time HTML attributes remains deferred.
  If added later, it should stay separate from generated wrapper constructor
  arguments and live in a distinct server/client command or update layer.

* User-facing binding-convention documentation deferment:
  user-facing documentation for the Shiny binding convention remains
  deferred.
  Future documentation should explain explicitly that action-like controls
  may bind as actions, action-with-payload controls such as `wa-dropdown`
  split the action trigger from the latest payload side channel, and other
  bindings may expose durable semantic state rather than raw browser event
  names.

* Semantic binding surface documentation deferment:
  user-facing documentation for the current semantic binding surface remains
  deferred.
  The current points that need explicit documentation are the native 0-based
  `wa-carousel` value contract, the selected-item-id contract plus
  id-omission warning behavior for `wa-tree`, the boolean `open` contracts
  for `wa-details`, `wa-dialog`, and `wa-drawer`, the active-tab-name
  contract for `wa-tab-group`, and the ordinary durable-value behavior now
  covered by the dedicated semantic harness apps.

* Warning and user-option documentation deferment:
  the package warning registry and related user-facing runtime options remain
  underdocumented for users and maintainers beyond code comments and runtime
  warnings.
  Future documentation should explain
  `options(shiny.webawesome.warnings = list(...))`, the currently known
  warning keys and suppression semantics, and the optional
  `options(shiny.webawesome.runtime_harness.log = TRUE)` development-only
  harness logging flag added during this session.

## Pitfalls, reminders, and pointers

* The browser-runtime layer is now component-owned. Future additions should
  continue that ownership model rather than rebuilding another shared
  monolithic browser test file.
* Harness apps are grouped by contract type:
  durable semantic values, semantic events, action/action-with-payload,
  tree-specific behavior, and presentational/runtime-presence.
  New browser-runtime tests should attach to the right contract app rather
  than to whatever app happens to be convenient.
* The app-side harness helpers are sourced local files.
  If new helper functions are added there, keep the sourced-helper lint
  suppressions narrow and only add one-line purpose comments where they
  clarify non-obvious intent.
* The visible harness state blocks are for humans first and for quick debug
  second.
  The browser tests should still prefer direct `shinytest2` `get_value()`
  or `wait_for_value()` assertions for real Shiny-contract checks.
* The old `representative-runtime` harness and `test-wa-browser-runtime.R`
  are intentionally gone.
  Future work should not resurrect them or drift back toward a single shared
  runtime fixture/test file.
