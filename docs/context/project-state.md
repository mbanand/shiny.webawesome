# Project State — Component Coverage Manifest Implementation

This document summarizes the current state of the shiny.webawesome development
workflow so that new development sessions can resume work without re-deriving
architecture decisions.

## Current Architecture Status

The repository uses a **generator-first architecture** for producing R/Shiny
wrappers around the Web Awesome component library.

Key principles:

- Component wrappers are generated from upstream metadata (`custom-elements.json`)
- Runtime assets are bundled from the upstream distribution
- Generated files must not be edited manually
- Build artifacts must be deterministic

## Build Pipeline

The project uses the following build pipeline:

clean → fetch → prune → generate → test → report

Generation produces:

- wrapper functions
- update functions
- bindings
- structured manifests

The **report stage** evaluates upstream coverage and API conformance.

## Manifest System

## Component Coverage Manifest

Component coverage tracks support for upstream Web Awesome components.

The final manifest:

build/manifests/component-coverage.yaml

is generated by merging:

1. upstream metadata (`custom-elements.json`)
2. discovered generated artifacts
3. policy annotations

Policy file:

dev/manifests/component-coverage.policy.yaml

## Merge Semantics

For each upstream component:

1. derive normalized component names
2. detect generated artifact existence
3. apply policy annotations when present

Rules:

- policy `status` overrides inferred status
- discovered artifact existence always reflects actual state
- if no policy entry exists:
  - wrapper exists → `covered`
  - wrapper missing → `unsupported`

Policy/discovery conflicts are allowed and are surfaced during the report stage.
